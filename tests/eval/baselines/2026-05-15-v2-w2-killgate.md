> ## Why v2 looks worse on the rubric — and why it isn't actually worse
>
> 1. **F-11 store queries all dropped to 0/7.** The agent has only `search_products` —
>    NO `search_stores` tool. So "מסעדות בתל אביב" correctly produces no tool call
>    and is labeled `intent=help`. v1 had a store search code path; v2 doesn't yet.
>    **This is W3 work, not a v2 regression.**
>
> 2. **F-03 needs_location dropped to 0/4.** The v2 endpoint doesn't surface a
>    `needs_location` flag — that comes from a future `clarify` tool. **Also W3.**
>
> 3. **brand_top_result unchanged.** Gemini IS calling search_products with
>    `brand='Sony'` — but the underlying search function doesn't strictly filter by
>    brand. The agent loop alone can't fix data quality. **W4 audit fixes land
>    brand filter + city normalization.**
>
> 4. **What v2 does correctly already:** 30/31 Hebrew product queries trigger
>    `search_products` with the right args (including all 3 single-brand-only
>    queries that v1 routed to clarify per F-09). Multi-turn comparison
>    (`sally-avi-comparison`) is harder to read in this baseline since v1
>    already passed it accidentally; v2 properly exited without re-searching.
>
> ## Recommendations going into W3
>
> - Add `search_stores` tool — unblocks F-11 (7 queries), F-03 (4 queries), and
>   most Sally store scenarios. Expected +30-40 pts on rubric overall.
> - Add `clarify` tool (or system-prompt rule) so the agent can return clarifying
>   questions for ambiguous opens (Sally Sarah scenario) without forcing a search.
> - Keep the brand_filter fix on the W4 docket — it's downstream of the agent loop.
>
> Bottom line: **v2 thesis confirmed — the agent loop on Gemini-2.5-flash is
> viable in Hebrew. Continue per the 11-week plan; no provider swap.**
