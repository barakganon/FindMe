# FindMe — Implementation Artifacts

This directory holds BMad story specs and the live sprint tracker.

## Files

| File | Purpose |
|---|---|
| [sprint-status.yaml](sprint-status.yaml) | Single source of truth for epic and story status. Edited by `/bmad-create-story`, `/bmad-dev-story`, and humans. |
| `<epic>-<story>-<slug>.md` | Per-story dev specs. For Epic 1 these are thin BMad-shaped indexes that point at [START_PROMPT.md](../../START_PROMPT.md) — see below. For Epics 3+ they're full specs generated by `/bmad-create-story`. |

## Epic 1 stories — read this first

[START_PROMPT.md](../../START_PROMPT.md) at the repo root is the authoritative
deploy plan. Phases 1–4 of that document map 1:1 to Stories 1.2–1.4. The
per-story files in this directory are **thin indexes** — they restate the
ACs and tasks in BMad's expected shape but defer all implementation details
to START_PROMPT.md.

This pattern keeps START_PROMPT.md as the single source of truth (no risk
of the BMad story files drifting out of sync) while still letting BMad's
auto-discovery work — the workflow strictly requires `<story_key>.md` files
to exist for `ready-for-dev` stories.

### Why Story 1.1 has no spec file

Story 1.1 (pre-deploy cleanup) was already executed in the 2026-05-03 session
before the BMad infra was fully wired. Its evidence lives in:
- [STATUS.md](../../STATUS.md) — Session 2026-05-03 (later) Phase 0 record
- Git history: commits `6792141`, `d61503b`, `8ad27a4`, merge `3a02b34`

Retroactively creating a story file would duplicate that evidence with no
value. The story is marked `done` in sprint-status.yaml.

### Working on Stories 1.2–1.4

Run `/bmad-dev-story` and it will auto-discover the next ready story
(currently 1.2). Open the discovered story file and read it as a thin
preamble, then execute the relevant phase of START_PROMPT.md. Update the
Dev Agent Record sections in the story file as you go.

## Epics 3+ — full BMad ceremony

Run `/bmad-create-story` normally. Those are multi-day code-heavy stories
(per-scraper investigation, new voucher network end-to-end, chain
enrichment rollout) where the upfront BMad ceremony — Dev Notes,
architecture compliance, source hints, previous-story intelligence —
genuinely prevents rabbit-holes.

## Status flow reminder

```
backlog → ready-for-dev → in-progress → review → done
```

- `backlog`: story exists only in [epics.md](../planning-artifacts/epics.md), spec file may exist as a future-dated stub
- `ready-for-dev`: spec file present in this directory, ready for `/bmad-dev-story`
- `in-progress`: dev actively working
- `review`: implementation done, awaiting code review
- `done`: shipped to master, verified
