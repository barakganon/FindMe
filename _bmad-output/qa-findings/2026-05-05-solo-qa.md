## BLOCK-tier issues (fix before any external user)

### F-01 / F-08 🔴 Search relevance broken on brand+category queries

**The single biggest UX problem.** "Most natural query" produces junk.

| Query | Top result | What user wanted |
|---|---|---|
| אוזניות סוני (Sony headphones) | עגילי סאני (Sani earrings) ₪140 | Sony headphones |
| Sony headphones | Edifier headphones | Sony headphones |
| Sony WH-1000XM5 | D-LINK IP CAM (mentions "Sony Exmor") | Sony WH-1000XM5 |
| שעון אפל (Apple Watch) | Apple Lightning earphones, then Apple Lightning cable | Apple Watch |

**Cause:** Hybrid search gives roughly equal weight to brand-token-match and category-token-match. When neither side has a direct hit (no exact "Sony" match in headphone descriptions, no exact "Apple" in watch descriptions), it falls back to whichever token appears in *anything*. Brand exact-match should be a hard filter or a heavy boost.

**Fix:** in [api/routes/chat.py](api/routes/chat.py) `_run_product_search`, when the parsed intent has a `brand`, add a strict filter on `products.brand ILIKE '%<brand>%'` for the candidate pool, OR boost the similarity score by 0.3-0.5 when the product brand matches.

**Effort:** 2-3 hours including tests.

### F-02 🔴 Intent parser non-deterministically returns clarify on clear queries

| Query | Run 1 | Run 2-4 (after caching) |
|---|---|---|
| כיסא ארגונומי לעבודה (ergonomic chair for work) | clarify ❌ | product_search ✅ |
| מסעדות בתל אביב (restaurants in Tel Aviv) | clarify ❌ | store_search ✅ |
| חנויות בגדים באזור שלי (clothing stores in my area) | clarify, needs_location=False ❌ | (consistent — really broken, not random) |

**Cause:** Non-deterministic Gemini output for borderline queries. First call lands on clarify ~10-20% of the time even on canonical queries. The intent cache (2 min TTL) then locks in whichever result the first call produced.

**Fix:** Set `temperature=0` on intent-parse calls, and tighten `INTENT_PARSER_SYSTEM` in [api/prompts.py](api/prompts.py) with explicit examples of common Hebrew phrasings.

**Effort:** 1-2 hours + manual prompt iteration.

---

### F-03 🔴 `needs_location` only matches "לידי" — synonyms break

| Query | Got | Expected |
|---|---|---|
| מסעדות לידי | needs_location=True ✅ | needs_location=True |
| חנויות בגדים באזור שלי | needs_location=False ❌ | needs_location=True |
| לידי, חנויות בגדים, באזור שלי, קרוב אלי | (all synonyms — only first works) | all should trigger GPS |

**Cause:** `INTENT_PARSER_SYSTEM` lists "באזור שלי" as a trigger but Gemini doesn't enforce it consistently. Either add deterministic post-processing (regex-match common phrases after Gemini parse, force `needs_user_location=True`) or expand the prompt with examples.

**Fix:** Post-process the `ParsedIntent` in [api/routes/chat.py](api/routes/chat.py): if user message matches `/(?:לידי|לי|באזור שלי|קרוב אלי|קרוב|פה|כאן)/`, force `needs_user_location=True` regardless of what Gemini said. Belt-and-suspenders.

**Effort:** 30 min including regex tests.

### F-04 🔴 Reply text contradicts results

**B1.09:** Returns 10 product results but reply text starts "לא מצאנו את ה-" (we didn't find the…). Self-contradiction.

**Cause:** Response composer composes a "no results" message based on weak similarity scores, but search still surfaces 10 candidates. Composer + searcher disagree on what counts as "found".

**Fix:** in [api/routes/chat.py](api/routes/chat.py) `_compose_response`, only let the LLM say "didn't find" when `len(product_results) == 0`. Otherwise force a positive opener.

**Effort:** 30 min.

---

### F-09 🔴 Single brand name returns clarify

**B2.05:** "סמסונג" alone → clarify. Should browse all Samsung products.

**Cause:** Intent parser treats single brand as "incomplete". For BuyMe users it's a totally legitimate query — "what Samsung stuff is available?".

**Fix:** Add explicit example to `INTENT_PARSER_SYSTEM`: a single brand token = product_search with `brand=<that token>` and `product_query=null`.

**Effort:** 30 min including prompt iteration.

### F-11 🔴 City matching uses BuyMe region buckets instead of city names

The shocker. Tel Aviv has 1 store with `city='תל אביב-יפו'` — but **407 stores** with `city='ת"א והסביבה'` (BuyMe's own region naming) and **450** geocoded inside the TLV bounding box.

| Bucket value | Count |
|---|---|
| ת"א והסביבה | 407 ← this *is* the TLV bucket |
| מרכז | 260 |
| סניפים בפריסה ארצית | 94 ← nationwide chains |
| צפון | 74 |
| השרון והסביבה | 48 |
| ירושלים והסביבה | 47 |
| ... | |

User types "תל אביב" → SQL `city ILIKE '%תל אביב%'` → 1 row.

**Three escalating fixes (do them all):**

1. **Synonym map** in store_search: when user says `תל אביב`, expand the SQL to also match `ת"א`, `תא`, `תל-אביב`, `יפו`. Same for Jerusalem (ירושלים → י-ם, ירושלים והסביבה), Haifa, etc.
2. **Geo fallback** when string match returns < 5 results: if we have user GPS or a city centroid, use lat/lng radius search (we already have geocoded coords for 1,076 of 1,236 stores).
3. **(Future)** A `city_normalized` column populated from a normalization pass — but the synonym map gets us 80% there immediately.

**Effort:** Synonym fix alone = 1-2 hours. Geo fallback = +2-3 hours. Total = ~half a day for a massive coverage improvement.

### F-06 🟡 English fragments leak into Hebrew replies

**B1.05 reply:** "חיפשנו 'spa' ומצאנו…" — used English word inside Hebrew sentence.
